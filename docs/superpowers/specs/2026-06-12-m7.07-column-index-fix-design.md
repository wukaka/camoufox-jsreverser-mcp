# M7.07 — Firefox 150 Column-Index Fix & objectInspector Live Restore

**Date:** 2026-06-12
**Branch:** `feat/m1-foundation`
**Predecessor:** M7.06 (`b1c50e8` — Firefox 150 attach + setBreakpoint protocol wiring)
**Successor:** unblocks `objectInspector` live regression; future M8 work on reverse-debug ergonomics builds on this.

## 1. Problem

After M7.06, `pauseController` correctly speaks the Firefox 150 RDP setBreakpoint protocol at the wire level. The `pauseController` live integration suite passes (set + remove returns no `DriverProtocolError`), but the **trigger-pause loop never fires**: a breakpoint is registered with the thread, the user code that should hit it runs, and no `paused` event arrives. As a result:

- `test/integration/capabilities/objectInspector.test.ts` is still `describe.skip`-ed.
- Any downstream capability that needs a live paused frame (`evaluateOnCallframe`, grip inspection, step) is non-functional against Firefox 150 in practice.

Root cause: Firefox 150's `thread.setBreakpoint` resolves the supplied `location.column` through `Debugger.Source.getPossibleBreakpoints`. The current code passes `idx + 1` from a substring search — an arbitrary text column that almost never lands on a legal breakpoint position. The thread silently drops the breakpoint (reply has no `error`, but the breakpoint never arms). This matches CDP-era best practice: clients are expected either to omit `column` or to snap the requested column to a legal position; older Firefox builds were more permissive and hid this from us.

## 2. Goals

1. Make `setBreakpointByText` / `setBreakpointByLocation` reliably arm a breakpoint on Firefox 150 that **actually triggers**.
2. Reach CDP-equivalent reverse-debug ergonomics:
   - default: precise, server-resolved location with an `actualLocation` echoed back to the caller (the CDP baseline);
   - opt-in: client-side "must stop in this column range" multi-hit auto-resume, for multi-statement / minified rows that the default would miss.
3. Re-enable the `objectInspector` live integration suite.
4. Preserve the public `BreakpointEntry` / `PauseController` API in a forward-compatible way (existing callers keep working, new fields are additive).

## 3. Non-Goals

- We do **not** add `actualLocation` reporting at the RDP wire level (Firefox 150 doesn't emit it; we synthesize ours from the `possibleBreakpoints` lookup).
- We do **not** change `evaluateOnCallframe`, `step_*`, `pause`/`resume`, or `freezeCurrent` behavior.
- We do **not** add multi-occurrence search (`occurrence: N`) in this milestone; the field is sketched in types but unused, and we deliberately defer the implementation to a later milestone if needed.
- We do **not** refactor `ScriptCache` beyond adding the per-source `possibleBreakpoints` index.
- We do **not** address `wsObserver` / `dials.setScriptingEnabled` / `dials.setBypassCSP` skipped suites — those have independent root causes tracked under M7.09 and upstream Firefox BiDi.

## 4. Design

### 4.1 Two-mode capability surface

`PauseController` gains an opts parameter on the two setBreakpoint entry points. **Default behavior is precise snap (mode B)**. Opt-in `columnTolerance > 0` adds multi-hit auto-resume (mode C).

```ts
// src/capabilities/types.ts (additive)
export interface BreakpointOptions {
  /** Width (in columns) of an acceptable hit window around requestedColumn.
   *  - undefined or 0  → accept whatever actualColumn we snapped to (CDP-default).
   *  - > 0             → in onPausedEvent, if frame.where.column is outside
   *                      [requestedColumn - tolerance, requestedColumn + tolerance],
   *                      auto-resume up to MAX_SKIPS times before forcing accept. */
  columnTolerance?: number;
}

export interface BreakpointEntry {
  bpId: string;
  bpActor: string;          // unchanged: threadActor on Firefox 150
  sourceActor: string;
  sourceUrl: string;
  line: number;             // unchanged: requested line
  column?: number;          // unchanged: requested column, if any

  // M7.07 additions (all optional, all populated when applicable):
  requestedColumn?: number; // mirror of `column`, kept for clarity in entry shape
  actualLine?: number;      // server-snapped line (usually === line)
  actualColumn?: number;    // snapped column from possibleBreakpoints
  columnTolerance?: number; // forwarded from BreakpointOptions for multi-hit gate
}

export interface PauseController {
  // unchanged shape, opts added at the tail (back-compat):
  setBreakpointByLocation(
    sourceUrl: string, line: number, column?: number, opts?: BreakpointOptions,
  ): Promise<BreakpointEntry>;

  setBreakpointByText(
    text: string, sourceUrl?: string, opts?: BreakpointOptions,
  ): Promise<BreakpointEntry>;

  // unchanged below…
}
```

### 4.2 `setBreakpointByLocation` flow

```
1. require threadActor (unchanged)
2. resolve source via thread.sources (unchanged), find {actor,url}
3. positions = await getPossibleBreakpointsForLine(threadActor, sourceActor, line)
   - cached per (sourceHash, line) in ScriptCache; fill on miss
4. if column !== undefined:
     actualCol = snapColumn(positions, column)
   else:
     actualCol = positions[0]?.column         // first legal position on the line
5. send thread.setBreakpoint with location =
     { sourceUrl, line, ...(actualCol !== undefined ? {column: actualCol} : {}) }
6. on error → BreakpointUnresolvedError (unchanged)
7. build BreakpointEntry with requestedColumn/actualLine/actualColumn/columnTolerance
8. if columnTolerance > 0, record in breakpoints map for the paused-event filter
```

**`snapColumn(positions, desiredCol)`**
- if `positions` empty → return `undefined` (caller will omit `column`).
- else find the position with smallest `|p.column - desiredCol|`; tie-break toward `p.column >= desiredCol` (the next legal stop is generally what the caller meant by "after this character").

**`getPossibleBreakpointsForLine`**
- sends `{ type: 'getPossibleBreakpoints', start:{line, column:0}, end:{line:line+1, column:0} }` to the **source actor** (Firefox 150 hosts this on `Debugger.Source.prototype`-backed actors).
- on protocol error or empty payload → return `[]` and let the caller fall back to "omit column".
- caches `Position[]` against `ScriptCache.get(sourceUrl).hash` so re-arming the same script is one round trip.

### 4.3 `setBreakpointByText` flow

Unchanged for the search side:
```
for each cached script (filtered by sourceUrl if given):
  for each line i:
    idx = line.indexOf(text)
    if idx >= 0:
      return setBreakpointByLocation(cached.url, i+1, idx+1, opts)
```
Only difference: `idx + 1` is now the **requested** column. The server-side snap inside `setBreakpointByLocation` produces the actual column. `opts.columnTolerance` is forwarded as-is.

### 4.4 Multi-hit auto-resume (opt-in, mode C)

In the persistent `onPausedEvent(f: PausedFrame)`:

```
1. Determine hitLine/hitColumn from (f.frame ?? f.currentFrame).where
2. For each breakpoint entry where:
     - entry.columnTolerance && entry.columnTolerance > 0
     - entry.actualLine === hitLine (or entry.line if actualLine missing)
   compute:
     ok = |hitColumn - entry.requestedColumn| <= entry.columnTolerance
3. If any entry says "not ok" AND no entry says "ok":
     entry.skipCount = (entry.skipCount ?? 0) + 1
     if entry.skipCount < MAX_SKIPS (= 8):
        rdp.call(threadActor, { type:'resume' })  // do NOT emit __pauseController.paused
        return
     else:
        // give up; treat as a normal hit with forcedAccept=true on the relayed event
4. recordPauseCtx(f); ee.emit('__pauseController.paused', f)
```

Notes:
- The 8-skip ceiling protects against tight loops (VMP dispatch). It is a safety net, not a performance feature; callers who need to scan many positions should narrow `columnTolerance` instead.
- `skipCount` lives on the in-memory `BreakpointEntry` only; it resets on `removeBreakpoint`.
- We deliberately do **not** suppress the event when `entry.columnTolerance === 0 || undefined` — that is the CDP-default mode.

### 4.5 `removeBreakpoint`

Send `{ type: 'removeBreakpoint', location: { sourceUrl, line: actualLine ?? line, ...(actualColumn !== undefined ? {column: actualColumn} : {}) } }`. Using the actual location guarantees the server can match the breakpoint it stored; if we sent the original `requestedColumn` we'd miss after the snap.

### 4.6 `ScriptCache` extension

```ts
// src/session/caches.ts
interface ScriptEntry {
  id: string; url: string; source: string; hash: string;
  // NEW:
  possibleBreakpoints?: Map<number /* line */, Array<{line:number; column:number}>>;
}
```

- Lazy: `get()` returns the entry; `pauseController` calls `cache.getOrComputePositions(url, line, fetchFn)` which fills and returns.
- Invalidated when `put()` is called for the same URL with a different `hash`.

### 4.7 Tool surface

`src/tools/debugger/set_breakpoint.ts` and `set_breakpoint_on_text.ts` extend their zod schemas with:
```ts
columnTolerance: z.number().int().min(0).max(1000).optional()
```
Schemas remain backward-compatible. The handler forwards as `opts.columnTolerance` to the capability. No new tool. No change to `remove_breakpoint`, `pause`, `resume`, `step_*`, `evaluate_on_callframe`.

## 5. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| `getPossibleBreakpoints` payload shape differs on Firefox 150 from what we expect | Guard with `try`/`catch`; on any failure, return `[]` and fall back to "omit column". Live test surfaces shape problems immediately. |
| Multi-hit in VMP dispatch loop floods the RDP socket | `MAX_SKIPS = 8` ceiling; documented guidance "use `columnTolerance: 0` (default) in VMP scenarios". |
| Side-effect leakage in `columnTolerance > 0` mode (statements before the desired column have already executed by the time we stop) | Documented in capability-level JSDoc and README so reverse-engineers know multi-hit is **not** a step-back primitive. |
| Stale `possibleBreakpoints` cache after page reload re-publishes the same URL with different bytes | Re-keyed by `ScriptEntry.hash`; `ScriptCache.put()` with a new hash drops the cached positions for that URL. |
| Adding `opts?` to public methods could surprise existing callers | Last-positional optional parameter; default value preserves prior behavior. Existing call sites in `set_breakpoint*` tools updated in the same change. |

## 6. Out-of-Scope Follow-ups

- `occurrence: N` on `setBreakpointByText` for "nth match" — sketched in BreakpointOptions only if it slips back in scope; otherwise left for M7.10 / M8.
- Exposing `actualLine`/`actualColumn` on tool replies so LLM callers see the snapped location. Trivial follow-up once the capability surface is in place.
- Step-into / step-over column awareness — Firefox 150's `next` step already targets the next legal position, so no immediate change needed.
