# M7.09 — Worker stealth: push firefox-default preset into worker contexts

**Date:** 2026-06-12
**Branch:** `feat/m1-foundation`
**Predecessor:** M7.08 (`591bc5b` — README + CLAUDE.md refresh)
**Successor:** M7.10 will expose this through an MCP tool.

## 1. Problem

`stealth.applyPreset('firefox-default')` currently only injects into the main world via BiDi `script.addPreloadScript`. Firefox's BiDi addPreloadScript does NOT propagate to worker realms. As a result:

- A page that runs anti-bot detection inside a **dedicated worker** (a common reverse-engineering target — sign / encrypt / fingerprint code routinely lives in a worker) reads `navigator.webdriver === true`, sees an unmasked `Function.prototype.toString`, and trips detection that the main-world stealth was supposed to suppress.
- `Session.init`'s `stealth === 'auto'` path silently leaves workers exposed.
- `preloadInjector.addToWorker()` already exists (it post-start `script.callFunction`s a payload into a worker realm) and `hookRegistry.inject({ target: 'all-workers' })` already uses it for user-injected hooks, but `stealth` capability never calls it.

M7.09 adds an opt-in `stealth.applyPresetToWorkers()` method that walks the current worker list, post-start-injects the preset payload into each dedicated / shared worker, and optionally subscribes to `workerTopology.onWorkerAvailable` to keep new workers covered.

## 2. Goals

1. A caller can run `stealth.applyPresetToWorkers('firefox-default')` and have the same payload as `applyPreset('firefox-default')` reach every existing dedicated + shared worker.
2. By default (`watch: true`), subsequently spawned workers automatically get the same payload.
3. Per-worker failures don't abort the batch; the caller receives a structured report (`injected[]`, `failed[]`) along with explicit `injectedAt: 'post-start'` so they know workers' prologue code has already run.
4. Service-workers are deliberately excluded in this milestone — they have different lifetime semantics (cross-tab persistence after `register`) that need a separate decision.
5. `--stealth=auto` (the Session.init default path) stays main-world-only. Worker stealth is an explicit second call so behavior does not change for users who didn't ask for it.
6. No MCP tool surface in this milestone. M7.10 owns the tool exposure for stealthHook + this method.

## 3. Non-Goals

- No pre-start worker injection. The BiDi gap (`addPreloadScript` doesn't reach workers) is upstream and out of scope. Document the post-start limitation, do not try to work around it with `Worker.prototype.constructor` proxying in M7.09.
- No service-worker support. Add a follow-up method `applyPresetToServiceWorkers` later if needed.
- No change to the `firefox-default` preload payload. Worker compatibility (e.g. `window.chrome` stub in a `WorkerGlobalScope`) relies on the payload's existing try/catch wrappers — features that don't apply silently no-op.
- No new MCP tool, no CLI flag, no README change. Tool exposure is M7.10.
- No `stealth.removePresetFromWorkers()` in M7.09. There is no `preloadInjector.removeFromWorker()` today; uninstall would require either re-loading the worker or shipping a counter-payload — out of scope.
- BiDi-only mode (no RDP) cannot push new-worker events; `watch: true` becomes effectively a no-op for future workers. This is documented, not patched.

## 4. Design

### 4.1 Type additions (in `src/capabilities/types.ts`)

```ts
export interface WorkerStealthInjection {
  /** Worker realmIds that successfully received the preset payload. */
  injected: string[];
  /** Workers we attempted but failed; reason is the underlying error message. */
  failed: { realmId: string; reason: string }[];
  /** When the injection ran relative to the worker's lifecycle. Always 'post-start'
   *  in this milestone — BiDi does not propagate addPreloadScript to worker realms,
   *  so the worker's prologue (including any early navigator.webdriver reads or
   *  Function.prototype.toString inspections) has already executed. */
  injectedAt: 'post-start';
  /** True when watching for new workers via workerTopology.onWorkerAvailable.
   *  In BiDi-only mode this stays true but the underlying subscription is a no-op
   *  (workerTopology.onWorkerAvailable does not push events on BiDi alone). */
  watching: boolean;
  /** Stops the onWorkerAvailable subscription. No-op when watching is false or
   *  the underlying subscription was already a no-op. Idempotent. */
  unwatch: () => void;
}

export interface ApplyPresetToWorkersOpts {
  /** Default true: subscribe to workerTopology.onWorkerAvailable and inject the
   *  preset into every dedicated/shared worker that appears after the call. */
  watch?: boolean;
}

// extension on existing Stealth interface:
export interface Stealth {
  // (existing methods unchanged)
  applyPresetToWorkers(
    presetName: string,
    opts?: ApplyPresetToWorkersOpts,
  ): Promise<WorkerStealthInjection>;
}
```

### 4.2 `stealth.applyPresetToWorkers` semantics

```
1. payload = PRESET_PAYLOADS[presetName]
   if undefined → throw Error('stealth: unknown preset ${presetName}')
2. if (!workers) → throw new StealthWorkersUnavailableError()
3. current = await workers.listWorkers()
   filtered = current.filter(w => w.type === 'worker')   // dedicated + shared
   (service-worker excluded by design)
4. injected = []
   failed = []
   alreadyInjected = new Set<string>()
   results = await Promise.allSettled(filtered.map(w =>
     preload.addToWorker(payload, w.realmId)
       .then(() => { injected.push(w.realmId); alreadyInjected.add(w.realmId); })
       .catch(e => { failed.push({ realmId: w.realmId, reason: String(e?.message ?? e) }); })
   ))
5. watching = opts?.watch ?? true
   unwatch = () => {}
   if (watching) {
     unwatch = workers.onWorkerAvailable((w) => {
       if (w.type !== 'worker') return                            // skip service-workers + others
       if (alreadyInjected.has(w.realmId)) return                  // de-dup vs. step 3
       alreadyInjected.add(w.realmId)
       void preload.addToWorker(payload, w.realmId).catch(() => {/* best-effort */})
     })
   }
6. return { injected, failed, injectedAt: 'post-start', watching, unwatch }
```

Concurrency notes:
- `Promise.allSettled` so one failure doesn't reject the batch.
- `alreadyInjected` is a `Set` shared between step 3 and step 5. It defends against the race where a worker first appears in `listWorkers()` and then pushes through `onWorkerAvailable` again (the order of the BiDi/RDP enumeration is not guaranteed against the watcher).
- The watch-mode injection in step 5 is fire-and-forget (`void … catch(() => {})`). Surfacing watch-mode failures up to the caller would require an event emitter the spec deliberately does not add — the caller can re-call `applyPresetToWorkers` to get a fresh report.

### 4.3 Factory signature change

`makeStealth(preload)` → `makeStealth(preload, workers?)`.

- `workers` is optional. If absent (e.g. early Session bootstrap before workerTopology is wired), `applyPreset` and `applyPresetToWorkers` behave as:
  - `applyPreset` — unchanged.
  - `applyPresetToWorkers` — throws `StealthWorkersUnavailableError`.

### 4.4 Error class

In `src/capabilities/errors.ts`:

```ts
export class StealthWorkersUnavailableError extends Error {
  readonly code = 'StealthWorkersUnavailableError';
  constructor() {
    super('stealth: workerTopology not available — workers cannot be injected');
  }
}
```

If the project's `error-translator.ts` maps capability errors to MCP `ErrorReason`s, add a new reason or reuse a close existing one (e.g. `WorkersUnavailable` or `StealthNotConfigured`). Implementation step in the plan will inspect the current translator and pick the consistent mapping; the spec leaves this as "must integrate with the existing translator pattern."

### 4.6 Test plan

**Unit (`test/unit/capabilities/stealth.test.ts`)** — extend existing file with a new `describe('applyPresetToWorkers', …)`:

1. Unknown preset → rejects with `unknown preset`.
2. No workers, default opts → `injected: []`, `failed: []`, `injectedAt: 'post-start'`, `watching: true`, `unwatch` is a function.
3. Two workers, both succeed → `injected.length === 2` and contains both realmIds; `failed: []`; `preload.addToWorker` called twice with `(payload, realmId)`.
4. One succeed + one reject → both arrays length 1; `failed[0].reason` contains the mock error message.
5. Mix of types → service-worker entries in `listWorkers()` output are filtered out (no `preload.addToWorker` call for them).
6. `watch: true` default → `workers.onWorkerAvailable` is invoked; when the registered callback fires with `{ type: 'worker', realmId: 'r3' }`, `preload.addToWorker(payload, 'r3')` is invoked; when it fires with `{ type: 'service-worker', realmId: 'r4' }`, no extra call.
7. `watch: false` → `workers.onWorkerAvailable` is NOT called; `watching: false`; `unwatch` no-op.
8. `unwatch()` after a watch-mode subscription → subsequent watcher pushes do NOT trigger `addToWorker`. (Achieved by invoking the unsubscribe function that `workers.onWorkerAvailable` returns; the mock should track this.)
9. Re-injection guard → `listWorkers()` returns `[w1]`; immediately after batch completes, the `onWorkerAvailable` callback fires with the SAME realmId — `addToWorker` is NOT called a second time.
10. `makeStealth(preload, undefined)` then call `applyPresetToWorkers` → rejects with `StealthWorkersUnavailableError`.

**Integration (`test/integration/capabilities/stealth.test.ts`)** — add one new case:

### 4.7 Migration / back-compat

`Stealth` interface adds a method (non-breaking for type consumers — TypeScript's structural type lets old implementations remain valid only inside callers that don't reach for the new method; production code is the only `makeStealth` factory and is updated in lockstep). No callers rely on the old single-arg `makeStealth` signature outside Session.init; the unit test for `applyPreset` constructs `makeStealth(preload)` with the second arg omitted, which the new signature still accepts.

## 5. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| post-start injection means worker prologue already ran; pre-prologue reads of `navigator.webdriver` slip through | `injectedAt: 'post-start'` is a fixed-value field in the return type so callers see the limitation explicitly. README + spec call this out. Workaround (e.g. proxying `Worker.prototype.constructor` from the main world) is deferred. |
| BiDi-only mode silently fails to push new workers under `watch: true` | `watching: true` still returned, with spec text "BiDi-only mode: the underlying onWorkerAvailable is a no-op." `workerTopology.onWorkerAvailable` in BiDi factory already returns an unsubscribe no-op (existing code at `src/capabilities/workerTopology.ts:23`). |
| firefox-default payload contains main-world-only fragments (`window.chrome` stub) that throw in a worker scope | The payload's IIFEs are individually try/catch-wrapped per existing stealth-script convention. Inapplicable features silently no-op. Add an integration assertion for the things that DO apply in worker scope: `navigator.webdriver === false`, `Function.prototype.toString` masking. |
| onWorkerAvailable callback fires for a realm that has been torn down between RDP push and our addToWorker call | `preload.addToWorker` already throws on `scriptHost.callFunction` exception; in watch mode we swallow with `.catch(() => {})`. Surfaced via spec note "watch-mode failures are silent; re-call applyPresetToWorkers for a fresh report." |
| Race between listWorkers() and onWorkerAvailable pushing the same realmId | `alreadyInjected: Set<string>` guards against double-injection. Unit test #9 locks this in. |
