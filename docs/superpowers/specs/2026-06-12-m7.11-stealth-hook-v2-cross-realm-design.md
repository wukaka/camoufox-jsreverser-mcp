## 1. Problem

Live measurement against CreepJS (https://abrahamjuliot.github.io/creepjs/) with a single `inject_stealth_hook(window.fetch)` call installed produces three concrete leaks:


|--------------------------------|----------|----------|------------------|

Evidence: `stealth-evidence/stage{3,4}-creepjs-*.{png,json}`.

Three root causes are now confirmed by direct DOM probe (`stealth-evidence/stage4-probe.json`, captured via `evaluate_script`):

1. **`Function.prototype.toString` override is realm-local.** The preload's IIFE patches `Function.prototype.toString` on the **top realm only**. Same-origin iframes (and `about:blank` children spawned by the page) get their own `Function.prototype`. Detection scripts that capture `iframe.contentWindow.Function.prototype.toString.call(iframe.contentWindow.fetch)` see the rewritten body verbatim ŌĆö variable names `__capture`, `__sample`, `channel: "fetch_calls"`, the channel emit identifier, and the function shape (multi-line, leading `var __args = arguments;`).
2. **Wrapper source body is itself fingerprintable.** Even if cross-realm masking is added, the rendered wrapper exposes a stable code shape with literal tokens (`__capture`, `__sample.threw`, `Reflect.apply(__original, this, __args)`, the channel-name string literal). A detector with a `__capture` substring lookup wins.
3. **Globals leak the toolchain.** `Object.defineProperty(globalThis, '__sh_mask__', { enumerable: false, configurable: false })` does **not** hide the key from the `in` operator, `Reflect.has`, or `Object.getOwnPropertyNames(globalThis)`. Anti-bot stacks routinely probe known stealth-tool global names. Same for `__sh_timing__`.

M7.11 closes these three at the renderer level. No new MCP tool. No new capability. The fix lives entirely inside `stealthHook` and the preload-injection path.

## 2. Goals

1. CreepJS `stealth%` returns to `0` and `Resistance.extension` returns to `unknown` after `inject_stealth_hook(window.fetch)` is installed on a fresh page.
2. `iframe.contentWindow.fetch.toString()` reports `"function fetch() { [native code] }"` for **same-origin** and **about:blank** iframes that exist at install time and that are created after install time.
3. `globalThis` exposes no enumerable, ownable, or `in`-detectable property whose key contains the substring `__sh_`, `__mcp_`, `stealth`, `hook`, or any of our channel emit names.
4. The rendered wrapper source, when read back via `Function.prototype.toString` in any path that bypasses the mask, contains **no literal token** specific to this implementation (no `__capture`, no `__sample`, no `channel` literal, no `Reflect.apply`-pattern that screams "wrapper"). The leaked form should resemble a plain `function name() { [native code] }` to every realm that has been masked, and a one-statement anonymous-shaped function to every realm that hasn't.
5. No regression for `mcp__JSReverser-Strong-MCP__inject_preload_script` based workflows. Existing M7.10 unit tests pass without change to their expected channel mechanics.

## 3. Non-Goals

- **No worker-realm hook injection.** `inject_stealth_hook` stays main-world + same-origin iframes only, consistent with M7.10's stated scope. Worker hooks remain a separate future milestone.
- **No UA spoofing.** The Camoufox UA string `Camoufox/150.0.2-beta.25` leak is a Camoufox-binary concern, addressed via `set_user_agent` or Camoufox fingerprint presets, not via stealthHook.
- **No fingerprint randomisation.** stealthHook hides the *fact of wrapping*, not the underlying hardware/timezone fingerprint surface. Resistance.privacy/security/mode stay `unknown` either way.
- **No timing-neutraliser change.** M7.05's `renderTimingNeutraliser` is orthogonal; its global `__sh_timing__` is renamed under the same closure-isolation rule but no behaviour change.
- **No MCP tool surface change.** `inject_stealth_hook` schema, return shape, and channel semantics stay identical. M7.11 is a renderer-internals upgrade.

## 4. Design

### 4.1 Cross-realm `Function.prototype.toString` masking

A single MutationObserver-based installer runs once per top realm. It does three things:

1. **Initial sweep at preload time.** For every existing `HTMLIFrameElement` in the document at preload-IIFE execution time, if `frame.contentWindow` is accessible (same-origin or `about:blank`), patch `frame.contentWindow.Function.prototype.toString` with the same closure-captured mask map already used on the top realm. The mask map is **shared by reference** so that a wrapped function registered in the top realm produces the same masked string in every nested realm.
2. **Future iframes.** A `MutationObserver` on `document` watches for added `HTMLIFrameElement` nodes. On addition, register a one-shot listener: `frame.addEventListener('load', ...)` plus an immediate inline check (some iframes synchronously expose a `contentWindow` before `load`). On both signals, run step 1 on the new frame.
3. **`document.open` reset guard.** A page that calls `document.open()` blows away the document and may regenerate `Function.prototype.toString`. We re-arm the override on the next microtask if the override has been replaced. Detected by storing the override function reference in a WeakRef and comparing on each first-toString-call within a `Reflect.has`-protected path.

The mask map lives in a `WeakMap` referenced exclusively by closures in the renderer-emitted IIFE. No global key, no `__sh_mask__`.

Edge cases ŌĆö explicit:

- **Cross-origin iframes**: `contentWindow.Function` access throws SecurityError. Caught and ignored; the page already cannot read those frames' fetch from JS, so detection cannot trivially compare. Documented as a known limitation.
- **Sandbox iframes without `allow-same-origin`**: same behaviour as cross-origin; cannot patch; documented.
- **Web Workers**: out of scope (M7.10 already says "main-world only").

### 4.2 Wrapper source cleansing

The current renderer produces multi-line wrapper bodies containing identifiers `__args`, `__sample`, `__capture`, `__ret`, plus the channel-name string literal. We replace this with a **single-line, minified** template that contains:

- Identifiers picked from a fixed set of one-character names (`a`, `b`, `o`, `d`, `k`, `p`, `e2`) ŌĆö no substring matches against any well-known stealth-toolchain identifier (no double-underscore-prefixed names).
- **No literal channel name in the wrapper body.** The channel name is passed in via a closure-captured parameter that the IIFE installs as a captured constant ŌĆö readable only via `Function.prototype.toString` on the closure body of the IIFE, never on the wrapped function returned to the page.
- **`Reflect.apply` replaced** with `o.apply(this, a)`, which is the same shape every Babel-transpiled function uses. Detector substring rules for "Reflect.apply" no longer trip.
- **No `__capture.indexOf(...)` runtime branches.** Render-time decides which capture branches to emit at all; runtime sees only the branches enabled at install time, each inlined as `try{b.args=...}catch(e){}` form.

**Preserved: sample object keys.** The emitted sample's keys (`args`, `ret`, `target`, `threw`, `stack`, `channel`, `ts`, `thisArg`) are part of the M7.10 channel-payload contract consumed by `get_hook_data`. They stay verbatim. The substring `channel:` will still appear in the wrapper toString, but as an object-literal key it's indistinguishable from common business code and not a stealth-tool signature.

The resulting wrapper, before mask is applied, is shape-equivalent to:

```javascript
function fetch(){var a=arguments,b={channel:k,ts:Date.now(),target:p};try{b.args=[].slice.call(a)}catch(e){}var d;try{d=o.apply(this,a)}catch(e){b.threw=String(e);try{e2(b)}catch(_){}throw e}b.ret=d;try{e2(b)}catch(_){}return d}
```

ŌĆ”and is then masked to `"function fetch() { [native code] }"` by ┬¦4.1 in any realm that has been visited.

A detector running `iframe.contentWindow.Function.prototype.toString.call(iframe.contentWindow.fetch)` *before* ┬¦4.1's MutationObserver fires (i.e., a synchronous read from a freshly-created iframe whose load event hasn't fired yet) sees the cleansed source above ŌĆö no `__capture` / `__sample` / `__args` / `__ret` / `Reflect.apply` identifier tokens, no `fetch_calls` literal. `channel:` and `args:` survive as object keys; they look identical to common business-logic event payloads and are not a stealth-toolchain signature. That is the worst case M7.11 must accept; the best case is `function fetch() { [native code] }` everywhere.

### 4.3 Closure isolation: no `__sh_*` globals

Both `__sh_mask__` and `__sh_timing__` move into closure-captured locals inside the preload IIFE. The IIFE is wrapped in `(function(){...}())` and never assigns to `globalThis`.

**Cross-IIFE behaviour (resolved with degradation):** Two `inject_stealth_hook` calls within the same session each render their own independent preload payload. The second IIFE's override of `Function.prototype.toString` replaces the first IIFE's override; wraps registered by the first payload then fall back to the unmasked `realFnToString` path and reveal their wrapper source.

We considered (and rejected) two alternatives for sharing the mask map across IIFEs:

1. **`Symbol.for('mcp.sh.toString.map.v1')` anchor on `Function.prototype.toString`.** Enables cross-IIFE map sharing, but `Object.getOwnPropertySymbols(Function.prototype.toString)` returns the symbol unconditionally (the `enumerable:false` descriptor does NOT hide own-symbols from this API), and `Symbol.keyFor` returns the literal string `'mcp.sh.toString.map.v1'`. A detector that runs `Object.getOwnPropertySymbols(Function.prototype.toString).map(Symbol.keyFor).filter(Boolean)` immediately fingerprints the toolchain. **Strictly worse** than the degradation.
2. **Private `Symbol()` anchor.** Each IIFE creates its own symbol, so cross-IIFE sharing is impossible by construction ŌĆö same outcome as the degradation, but with the extra cost of an attached-property leak (still visible via `getOwnPropertySymbols`, even if `Symbol.keyFor` returns undefined). **Strictly worse** than no attachment at all.

Therefore the design accepts the degradation: ┬¦4.2's wrapper-source cleansing already ensures that when a first-IIFE-registered wrap's `toString` falls back to `realFnToString`, the leaked body carries no stealth-toolchain identifier tokens (`__capture` / `__sample` / `__args` / `__ret` / `__emit` / `Reflect.apply` / channel-name literal). The cleansed wrapper looks like generic instrumented code, not a fingerprint of this toolchain.

`__sh_timing__` becomes a similar closure-local: the IIFE skips re-install if `performance.now` is already non-native (detected by checking `realCall.call(realFnToString, performance.now)` for a non-native body). No anchor property is added to `performance.now`.

### 4.4 Channel emit name

`__mcp_emit_<hex>` is the channel-emit global injected by `inject_stealth_hook` (it's how the wrapped function ships samples back to the MCP server). This is **a real global and must stay one** ŌĆö BiDi `script.callFunction` with `arguments` is the only path back, and that requires a globally-reachable function in the page realm.

Mitigations:

1. **Random hex per call** is already in place ŌĆö the global name carries `<hex>` so the literal `__mcp_emit_` is not enough; a detector would have to regex `__mcp_emit_[0-9a-f]+` against `Object.getOwnPropertyNames(globalThis)`.
2. **Hide via `Object.defineProperty` with `enumerable:false` AND `configurable:false`**. (Currently the descriptor is the default ŌĆö i.e., enumerable.) Combined with `delete` not working (non-configurable), this removes the key from `Object.keys` and `for..in`. It does **not** remove it from `Object.getOwnPropertyNames` ŌĆö that's a JS-spec limitation.
3. **Document the residual leak**: `Object.getOwnPropertyNames(globalThis)` regex on `__mcp_emit_` is a known false-positive vector. Recommend that high-risk targets call `remove_hook` (not yet exposed for stealthHook ŌĆö see "Followups") before any DOM read by a detector script.

### 4.5 No new channel, no new error code, no API change at the MCP tool layer

`inject_stealth_hook` and `inject_stealth_to_workers` keep their schemas. Return shape stays identical. The only difference an LLM observes is that CreepJS no longer flags the install.
