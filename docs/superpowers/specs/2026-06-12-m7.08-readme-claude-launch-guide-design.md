# M7.08 — README + CLAUDE.md Launch Guide (Camoufox-first)

**Date:** 2026-06-12
**Branch:** `feat/m1-foundation`
**Predecessor:** M7.07 (`7872a49` — column-index fix + objectInspector live restore)
**Scope:** Documentation only. No code changes.

## 1. Problem

After M7.01 the launcher became Camoufox-only (geckodriver-fronted, raw `firefox --remote-debugging-port` is no longer supported), but the docs were never refreshed. Today:

- `README.md` still describes the install as "Firefox (stable or ESR) + geckodriver" and never mentions Camoufox, even though `src/index.ts:28` hard-codes `/Applications/Camoufox.app/Contents/MacOS/camoufox` as the default binary path.
- The CLI flag table omits `--camoufox-path`, `--geckodriver-path`, and `--user-agent`, all of which are real flags wired in `src/server/argv.ts`.
- `.env.example` lists `FIREFOX_PATH`, which the launcher does not read.
- There is no `CLAUDE.md` at the repo root, so when an LLM coding agent (Claude Code or similar) cold-starts on this repo it has no curated entry point to the 4-layer architecture, 22 capabilities, 14 tool groups, or the M1-M7 milestone history.
- Real gotchas on macOS — Apple Silicon `brew` installs `geckodriver` to `/opt/homebrew/bin` instead of `/usr/local/bin`, `--stealth=off` does **not** disable Camoufox's C++-level stealth, live tests don't self-skip on marionette handshake failures — are nowhere written down.

A first-time human contributor or a fresh LLM agent following the current `README` will either fail to launch or launch with hidden misconceptions.

## 2. Goals

1. A human developer on a fresh macOS machine can read `README.md` and bring up the server end-to-end, with the right binaries in the right places.
2. A Claude Code-class LLM agent that opens this repo cold can read `CLAUDE.md` and pick the right file to inspect for any reasonable question about architecture, capabilities, tools, milestones, or known gotchas — without re-deriving facts from scratch.
3. `.env.example` matches what the launcher actually reads.
4. No code changes. Doc accuracy is verified against the current source, not against historical assumptions.

## 3. Non-Goals

- Do not refactor `argv.ts` or `FirefoxLauncher.ts`. Stale code paths (e.g. unused `FIREFOX_PATH` resolution if any) stay; we only stop documenting them.
- Do not cover Linux / Windows install in depth. README links out to the upstream Camoufox and geckodriver pages and stops.
- Do not regenerate the tool catalogue from runtime introspection. Tool counts are inventoried by directory listing, which the code review step verifies.
- Do not add a contributor guide (CONTRIBUTING.md), code-style doc, or release-process doc — out of scope for M7.08.

## 4. Design

### 4.1 README rewrite

Keep the existing **Architecture**, **Project layout**, **Test layers**, and **License** sections as-is — they are still accurate. Rewrite three sections:

**Install** (replaces the current "Install" block, ~15 lines → ~30 lines)
- Node.js 20+ requirement.
- One sentence stating the supported stack: **Camoufox + geckodriver only**. Raw `firefox --remote-debugging-port` exposes CDP, not BiDi, and is unsupported.
- macOS detailed steps:
  1. Download Camoufox `.dmg` from <https://github.com/daijro/camoufox/releases>; install to `/Applications/Camoufox.app`. First launch: right-click → Open to bypass Gatekeeper.
  2. `brew install geckodriver`. Apple Silicon installs to `/opt/homebrew/bin/geckodriver`; Intel to `/usr/local/bin/geckodriver`.
  3. `npm install && npm run build`.
- A one-line "Linux / Windows: see [Camoufox docs](https://github.com/daijro/camoufox) and [geckodriver releases](https://github.com/mozilla/geckodriver/releases)" pointer. No platform table.
- A short callout box explaining what "stealth" means in this project: `firefox-default` preload payload (`src/stealth-scripts/`) layered on top of Camoufox's C++ stealth. `--stealth=off` only disables the preload; the Camoufox layer is always on.

**Configure** (rewrite the env-var table)
- Remove the `FIREFOX_PATH` row.
- Add `CAMOUFOX_PATH` and `GECKODRIVER_PATH` rows with the macOS defaults and a note that Apple Silicon must override `GECKODRIVER_PATH=/opt/homebrew/bin/geckodriver` or pass `--geckodriver-path`.
- Keep the four LLM rows as-is.

**Run** (rewrite the CLI flag table)
- Add `--camoufox-path <path>` and `--geckodriver-path <path>` rows.
- Add `--user-agent <ua>` row (overrides the leaky `Camoufox/<ver>` UA brand).
- Remove `--firefox-path` (argv.ts has no such case).
- Keep `--attach`, `--bidi-url`, `--rdp-port`, `--stealth`.
- Add a one-line note: `--stealth=auto` (default) installs the firefox-default preload during `session.init`; `--stealth=off` skips it but does not disable Camoufox.

### 4.2 `.env.example` sync

- Remove `FIREFOX_PATH=`.
- Add `CAMOUFOX_PATH=` with a comment: `# macOS default: /Applications/Camoufox.app/Contents/MacOS/camoufox`.
- Keep `GECKODRIVER_PATH=` with an additional comment: `# Apple Silicon: /opt/homebrew/bin/geckodriver`.
- LLM rows unchanged.

### 4.3 New `CLAUDE.md` at repo root

Target: 150–200 lines. Sections:

1. **One-line project framing** — Firefox 150 BiDi+RDP reverse-engineering MCP server, Camoufox-only stack.
2. **Quick start** — three-line launch (env var or `--camoufox-path`, `node build/src/index.js --stealth=auto`).
3. **Architecture diagram** — same 4-layer flow as README, with a sentence on why driver/capability/session/tool stays separated (so an agent knows which layer to touch for a given change).
4. **Drivers (3 entries)** — one line per driver: file path + responsibility + one gotcha (e.g. RDP needs synchronous greeting consume per M7.04).
5. **Capabilities (22 entries)** — one line per capability: file path + 5-10-word responsibility. Grouped roughly: BiDi-backed, RDP-backed, AST/LLM, infra.
6. **Tool groups (14 entries)** — one line per group: directory + tool count (sourced from `ls src/tools/<group>/ | wc -l - 1`) + one-sentence purpose.
7. **Session & caches** — Session singleton, dispatcher, 5 caches (ScriptCache, RequestPool, HookTable, WsTable, ConsoleRing), the pause context, session-scoped pref overrides.
8. **Milestone log (M1-M7)** — one sentence per milestone phase: M1 foundation, M2 BiDi caps, M3 RDP caps, M4 stealth+AST+LLM, M5 rebuild+evidence, M6 tests+CI, M7 Firefox 150 + Camoufox fitness. Followed by a short table of M7 sub-milestones (M7.01-M7.08) with one-line summaries.
9. **Where to look** — single-source-of-truth pointers:
   - CLI flags / argv shape → `src/server/argv.ts`
   - Entry / wiring → `src/index.ts`
   - Specs → `docs/superpowers/specs/`
   - Plans → `docs/superpowers/plans/`
   - Stealth preloads → `src/stealth-scripts/`
   - Live fixture pages → `test/integration/fixtures/pages/`
10. **Gotchas** — bulleted list:
    - `--rdp-port` auto-allocates a free port when omitted in launch mode; attach mode requires explicit `--rdp-port`.
    - Apple Silicon: brew installs geckodriver to `/opt/homebrew/bin`, not `/usr/local/bin`. Override via `--geckodriver-path` or `GECKODRIVER_PATH`.
    - `--stealth=off` only disables the preload layer; Camoufox C++ stealth stays on.
    - LLM tools return `LlmNotConfigured` when no key is set — they never throw. The remaining ~76 tools work without an LLM key.
    - Live integration suites (`test/integration/`) **do not** self-skip on marionette handshake failures; they throw. Run them only when Camoufox is installed.
    - Firefox 150 `setBreakpoint` requires `column` to land on a legal `Debugger.Source.getPossibleBreakpoints` position. `pauseController` handles this transparently (M7.07); see the spec under `docs/superpowers/specs/` if you need to extend it.

### 4.4 Content sourcing

To keep CLAUDE.md accurate without copy-paste drift:

- Capability list: from `ls src/capabilities/*.ts` minus `types.ts` / `errors.ts` (current count: 22).
- Tool group list: `ls src/tools/` (14 directories).
- Tool counts per group: `ls src/tools/<group>/ | wc -l - 1` (subtract `index.ts`).
- Milestone summaries: from `git log --oneline | grep '(M[0-9]\+\.[0-9]\+)'`.

These are the authoritative pulls during the implementation plan; the spec doesn't pre-bake the numbers because the writer must verify against the current tree at write time.

## 5. Acceptance Criteria

1. README "Install / Configure / Run" sections describe the Camoufox + geckodriver stack accurately and contain no references to `FIREFOX_PATH` or `--firefox-path`.
2. `.env.example` rows match the env vars actually read by `src/index.ts` (verified by `grep "process.env\[" src/index.ts src/server/argv.ts`).
3. CLI flag table contents match `src/server/argv.ts` switch cases 1-for-1.
4. `CLAUDE.md` exists at the repo root, lists 22 capabilities with correct paths, lists 14 tool groups with correct counts, and contains the milestone log section.
5. All link anchors resolve: Camoufox repo, geckodriver releases, internal spec/plan paths.
6. `pnpm -s exec tsc --noEmit` still clean (no code touched).
7. `pnpm -s exec vitest run test/unit` still 505/505 (no code touched).
8. Manual smoke: a reader following only README on a fresh macOS shell can run `node build/src/index.js --stealth=auto` to a working tools-list response.

## 6. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| External links rot (Camoufox releases page restructured, brew formula renamed) | Link to repo root pages, not pinned versions. README install steps describe the action ("download dmg from releases") rather than naming a file. |
| CLAUDE.md drifts when new milestones land | Section 8 explicitly says "consult `git log --oneline | grep '(M…)'` for the current set"; the table is a snapshot for first-load orientation, not a maintained registry. |
| Capability or tool counts shift in a future PR without doc update | Section 4.4 / acceptance criterion 4 forces the implementer to re-run the directory listings at write time. A follow-up CI lint can be added later (out of scope here). |
| Reviewer expects platform parity (Linux / Windows install) | Out of scope per user direction. README explicitly says "see upstream docs" and stops; CLAUDE.md mentions only macOS-tested defaults. |

## 7. Out-of-Scope Follow-ups

- A CI doc-lint that grep-checks README CLI flags vs `argv.ts` cases.
- A `CONTRIBUTING.md` covering branch / commit / PR conventions.
- Linux / Windows install docs once those platforms get verified.
- An auto-generated tool catalogue (sourced from `tool-registry`) — would replace the manual count maintenance in CLAUDE.md.
